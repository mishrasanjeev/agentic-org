1. Platform Overview

1.1 What Is AgenticOrg?

AgenticOrg is an enterprise AI platform that deploys AI virtual employees — autonomous agents that act, not just chat. Unlike copilot products that assist humans with suggestions, AgenticOrg agents independently execute business processes end-to-end: they file GST returns, reconcile bank statements, process payroll, triage support tickets, score marketing leads, and manage vendor contracts — all without human intervention unless a decision gate requires approval.

Each agent operates under a formal Standard Operating Procedure (SOP), follows domain-specific prompt templates, and can be promoted from shadow mode (observation-only) to active mode (full autonomy) only after passing six quality gates. Human-in-the-loop (HITL) checkpoints ensure that high-value or high-risk actions always receive human oversight.

1.4 Architecture Overview

AgenticOrg is built on a modern, open-source-only stack:

▸ Agent Runtime

LangGraph-based orchestration engine. Each agent is a stateful graph with nodes for tool calls, LLM reasoning, and HITL gates. Supports per-agent LLM selection (GPT-4o, Gemini 2.5 Flash/Pro, Claude Sonnet).

▸ Authorization

Grantex authorization framework. Fine-grained scopes per agent (e.g., finance:ap:write). Scope enforcement with audit logging. JWT + API key authentication.

▸ Protocol Layer

A2A (Agent-to-Agent) protocol for inter-agent communication. MCP (Model Context Protocol) server for tool-provider integration. Composio for 1000+ third-party app integrations.

▸ Data Layer

PostgreSQL with Row-Level Security (RLS). pgvector for RAG vector search. Redis for caching. All data encrypted at rest (AES-256) and in transit (TLS 1.3). Deployed on Google Cloud Platform.

10.5 Shadow Mode & Six Quality Gates

Every new agent starts in Shadow Mode, where it observes real transactions and proposes actions without executing them. To be promoted to Active, the agent must pass six quality gates:

  1. Accuracy Gate: Proposed actions match expected outcomes ≥95% of the time
  2. Compliance Gate: All actions comply with regulatory requirements and SOPs
  3. Cost Gate: LLM costs per action remain within budget thresholds
  4. Latency Gate: Response times meet SLA requirements
  5. Safety Gate: No PII leaks, no unauthorized data access, no hallucinated actions
  6. Human Review Gate: A human reviewer (admin or domain lead) signs off on promotion

13. Approvals (HITL)

The Approvals page (/dashboard/approvals) is the human-in-the-loop queue where agents escalate decisions for human review.

13.1 Approval Queue

The queue has two tabs: Pending (items awaiting decision) and Decided (historical decisions). A red badge shows the count of pending items. Items can be filtered by priority: Critical, High, Normal, Low.

13.2 Approval Card

Each approval item displays:

  • Title: Description of what needs approval
  • Agent: Which AI agent submitted the request
  • Priority: Color-coded priority badge
  • Context: Relevant data and the agent's reasoning
  • Expires At: Deadline for the decision (items auto-expire if not acted upon)
  • Actions: Approve, Reject, or Request More Info buttons
  • Notes: Free-text field for the reviewer's comments

13.3 Decision Flow

  1. Agent encounters a decision gate (e.g., invoice amount > INR 5L, or confidence score < threshold)
  2. Agent creates an approval request with full context and reasoning trace
  3. Item appears in the approval queue for the appropriate reviewer
  4. Reviewer examines the context and makes a decision (approve/reject)
  5. Decision is recorded in the audit trail with HMAC signature
  6. Agent receives the decision and proceeds (or halts) accordingly

13.4 Escalation Rules

Configurable escalation rules determine when items escalate to a higher authority:

  • If pending for >4 hours, escalate to department head
  • If pending for >24 hours, escalate to admin
  • Critical items are immediately visible to all admins

13.5 Approval History

The "Decided" tab maintains a complete history of all decisions, searchable by date range, agent, and decision type. Each entry includes the full reasoning trace and decision notes.

21.1 JWT Authentication

21.3 Row-Level Security (RLS)

PostgreSQL Row-Level Security ensures that each tenant's data is completely isolated at the database level. Even if application-layer security is bypassed, RLS prevents cross-tenant data access.

21.4 PII Masking

When enabled (Settings → PII Masking), the system automatically detects and masks personally identifiable information in:

  • Audit trail entries
  • Agent reasoning traces
  • Chat panel conversations
  • API responses (when requested)

Masked formats: Aadhaar (XXXX-XXXX-1234), PAN (XXXXX1234X), email (s****@example.com), phone (+91-XXXXXX1234).

21.5 Audit Trail (7-Year Retention)

The Audit Trail page (/dashboard/audit) provides an immutable, HMAC-signed log of all platform activities. Each entry records:

  • Timestamp (UTC with nanosecond precision)
  • Actor (user email or agent name)
  • Action performed
  • Resource affected (agent, workflow, connector, approval)
  • Outcome (success/failure)
  • IP address and user agent
  • HMAC signature for tamper detection

Enterprise plans retain audit data for 7 years, meeting SOC-2 and DPDPA requirements.

21.6 Scope Enforcement (Grantex)

The Scope Dashboard (/dashboard/scopes) visualizes the permission scopes assigned to each agent. The Enforcement Audit Log (/dashboard/enforce-audit) records every scope check, including denied actions.

Scope format: <domain>:<resource>:<action> (e.g., finance:invoice:write, hr:payroll:read).

21.7 SOC-2 Readiness

AgenticOrg implements all 5 SOC-2 Trust Service Criteria:

  • Security: Encryption at rest (AES-256) and in transit (TLS 1.3), JWT auth, RLS
  • Availability: 99.9% uptime SLA, multi-zone deployment, health monitoring
  • Processing Integrity: HMAC-signed audit trail, agent quality gates
  • Confidentiality: PII masking, data classification, credential vault encryption
  • Privacy: DPDPA compliance, data region selection, consent management

21.8 DPDPA Compliance

Compliant with India's Digital Personal Data Protection Act (2023). Features include: data localization (India region), consent tracking, right to erasure, data processing records, and breach notification workflows.

  4. Indexing: Embeddings are stored in pgvector for fast similarity search

27. Frequently Asked Questions

Q1: What makes AgenticOrg different from AI copilots like ChatGPT or Microsoft Copilot?

Copilots assist humans by generating text or answering questions. AgenticOrg agents act autonomously — they execute complete business processes (filing tax returns, reconciling bank statements, processing payroll) without requiring continuous human input. They operate under SOPs with HITL gates for safety.

Q2: How many agents can I run simultaneously?

The Free plan supports 3 active agents. Pro supports 25. Enterprise has no limit. You can configure fleet limits in Settings.

Q3: Can agents make mistakes? How do you prevent errors?

Agents start in Shadow Mode where they propose actions without executing. They must pass 6 quality gates before promotion to Active. Confidence floors trigger HITL escalation for uncertain decisions. Kill Switch provides instant emergency stop.

Q4: Is my data secure?

Yes. Data is encrypted at rest (AES-256) and in transit (TLS 1.3). PostgreSQL Row-Level Security isolates tenant data. PII masking is enabled by default. The audit trail is HMAC-signed and tamper-proof.

Q5: Can I use AgenticOrg with my existing accounting software?

Yes. Native connectors exist for Tally, QuickBooks, Xero, Zoho Books, SAP, and Oracle ERP. The Tally Bridge enables real-time sync with desktop Tally installations.

Q6: Does AgenticOrg support GST filing?

Yes. The GST Filing Agent automates GSTR-1, GSTR-3B, and GSTR-9 filing via the GSTN connector. It supports DSC-based signing and 2A reconciliation. Manual JSON upload is available as a fallback.

Q7: How does the CA Firms module work?

CA firms purchase the CA Pack add-on to manage multiple client companies from a single login. Each company has isolated data, dedicated agents, and compliance tracking. The Partner Dashboard provides a bird's-eye view of all clients.

Q8: Can I bring my own LLM?

Yes. AgenticOrg supports per-agent LLM selection including GPT-4o, Gemini 2.5 Flash/Pro, and Claude Sonnet. For air-gapped deployments, local LLM support is available.

Q9: How does billing work for Indian customers?

Indian customers pay in INR via PineLabs Plural hosted checkout. All 6 payment methods are supported: Cards, UPI, Net Banking, Wallets, Credit EMI, and Debit EMI. Toggle currency to INR on the Billing page and click Upgrade. International customers pay in USD via Stripe (Cards, Google Pay, Apple Pay). The CA Pack add-on is priced at INR 4,999/month per firm (currently on free 14-day trial).

Q10: What happens when an agent reaches its budget limit?

At 80% budget utilization, a warning is generated. At 100%, the agent is automatically paused. You can increase the limit from the agent detail page or wait for the next billing cycle.

Q11: Can I create custom workflows from natural language?

Yes. The Workflow Creator accepts plain English descriptions and generates multi-step workflows with appropriate agent assignments and HITL gates.

Q12: What is the SLA for uptime?

Enterprise plan customers receive a 99.9% uptime SLA. The platform runs on Google Cloud Platform with multi-zone redundancy.

Q13: How do I export data from AgenticOrg?

Reports can be exported as PDF or Excel. The API provides full programmatic access to all data. Audit trails can be exported for compliance purposes.

Q14: Is there a mobile app?

Currently, AgenticOrg is a web application optimized for desktop and tablet browsers. Mobile optimization is planned for a future release.

Q15: What languages does the platform support?

The platform supports English as the primary language. The i18n framework is built-in and will support Hindi and other Indian languages in a future release.

Q16: Can agents communicate with each other?

Yes. The A2A (Agent-to-Agent) protocol enables agents to delegate tasks, share context, and coordinate complex workflows across domains.

Q17: How do I monitor agent performance?

Use the Observatory page for real-time monitoring. Each agent's detail page shows cost, accuracy, latency, and error metrics. The CEO Dashboard provides the Agent Observatory feed.

Q18: What is Grantex?

Grantex is the authorization framework that controls what each agent can do. It uses fine-grained scopes (e.g., finance:invoice:write) to ensure agents only access resources they are authorized for.

Q19: Can I self-host AgenticOrg?

Yes. Docker Compose for single-server deployments, Helm charts for Kubernetes clusters, and air-gapped deployment guides are all provided.

Q20: How do I get support?

Free plan: Community support via GitHub. Pro plan: Email support. Enterprise plan: Dedicated support with SLA. Contact: sanjeev@agenticorg.ai.

Q21: Does AgenticOrg use any proprietary or AGPL-licensed components?

No. AgenticOrg is built entirely on open-source tools. There are no LangSmith, AGPL, or proprietary SaaS dependencies.

Q22: What is the Composio marketplace?

Composio is a third-party integration platform that provides pre-built connectors for 1000+ applications. It extends AgenticOrg's native 54 connectors with additional SaaS integrations.

G.3 Incident Response Procedure

  1. Detection: Automated monitoring detects anomalous access patterns, failed auth attempts, or data exfiltration signals.
  2. Classification: Incident is classified by severity: P1 (critical data breach), P2 (potential exposure), P3 (suspicious activity), P4 (policy violation).
  3. Containment: Kill Switch activated for affected agents. Compromised API keys revoked. Affected tenant isolated.

  5. Notification: Affected customers notified within 72 hours per DPDPA requirements. Regulatory bodies informed if required.

Appendix I: Best Practices

I.1 Agent Configuration Best Practices

▸ Start with Shadow Mode

Never deploy an agent directly to Active mode. Always start in Shadow to observe proposed actions and validate against expected outcomes for at least one full business cycle (e.g., one month for monthly processes).

▸ Set Appropriate Confidence Floors

Higher floors (0.90+) are safer but generate more HITL escalations. Lower floors (0.80) give more autonomy but increase risk. Start high and lower gradually as trust builds.

▸ Use Domain-Specific LLMs

Assign Gemini Flash for high-volume routine tasks (ticket triage, email classification). Use GPT-4o or Claude for complex reasoning tasks (contract review, financial analysis). This optimizes cost without sacrificing quality.

▸ Implement Least-Privilege Scopes

Grant agents only the Grantex scopes they need. An AP Processor should not have hr:payroll:write access. Review and prune scopes quarterly.

I.2 Workflow Best Practices

▸ Add HITL at Critical Junctures

Place human checkpoints before irreversible actions: filing tax returns, sending bulk emails, making payments, or modifying employee records.

▸ Use Schedule Triggers Wisely

Align scheduled workflows with business rhythms. Run bank reconciliation daily, GST filing monthly, and TDS quarterly. Avoid scheduling all workflows at the same time to prevent resource contention.

▸ Monitor Failed Runs

Set up alerts for workflow failures. Most failures are caused by expired connector credentials, API rate limits, or data format changes. Address root causes promptly.

▸ Test with Templates First

Use built-in workflow templates before building custom workflows. Templates encode best practices and include appropriate HITL gates.

I.3 Security Best Practices

  • Rotate API keys every 90 days. Use expiry dates when creating keys to enforce automatic rotation.
  • Enable PII masking. This is on by default; do not disable it unless you have a specific, documented reason.
  • Use strong passwords. Minimum 12 characters with uppercase, lowercase, numbers, and special characters.
  • Review the audit trail weekly. Look for unusual access patterns, failed authentication attempts, and scope violations.
  • Limit admin accounts. Follow the principle of least privilege. Most users should have domain-specific roles (CFO, CHRO, etc.), not admin.
  • Keep connectors updated. Re-authenticate OAuth connectors before tokens expire. Monitor connector health daily.
  • Back up Knowledge Base regularly. Export critical documents and agent configurations periodically.
  • Use the Kill Switch judiciously. It stops ALL agents immediately. Use agent-level pause for targeted interventions.

I.4 CA Firms Best Practices

  • Onboard one client at a time. Complete the full 6-step wizard and verify all connections before onboarding the next client.
  • Set up compliance calendar alerts. Configure 7-day, 3-day, and 1-day reminders for all filing deadlines.
  • Use partner self-approval wisely. Enable only for routine filings (monthly GSTR-3B). Require dual approval for annual returns.
  • Store DSC details in the credential vault. Never share DSC passwords via email or chat.
  • Review client health scores weekly. Address red-flagged clients (score <50) immediately to prevent compliance lapses.
  • Train article clerks on the approval queue. They should understand how to review agent-proposed filings before partner sign-off.
