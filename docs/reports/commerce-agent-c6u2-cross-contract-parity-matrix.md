# Commerce Agent C6U2 Cross-Contract Parity Matrix

## Status

This is an internal AgenticOrg planning document for Grantex Commerce V1 cross-contract parity. It does not approve production launch, real merchants, public discovery, checkout/payment creation, live payments, live Plural, provider calls, merchant private API calls, production config, production allowlists, cloud resources, protocol publication, external submission, certification, compliance, conformance, standardization, public-launch readiness, merchant approval, or production readiness.

AgenticOrg must call Grantex only for commerce. AgenticOrg must not call payment providers, Plural, merchant private APIs, or merchant existing systems directly. Any missing capability below remains a future implementation slice, not a hidden launch approval.

## AgenticOrg-Owned Follow-Up Slices

1. C6U3: Add cross-repo alias schema and refusal parity tests, especially `payment_get_status` passport preflight and error mapping.
2. C6U4: Implement buyer-safe product/source/freshness/price/tax/warranty/return projection once Grantex supplies the shared safe shape.
3. C6U5: Add shared public discovery state handling for AgenticOrg A2A/MCP/agent metadata and buyer-session gate checks.
4. C6U6: Add durable buyer session, consent handoff, expiry, revocation propagation, and channel cache invalidation.
5. Web buyer baseline: create a hidden/internal web buyer session using only Grantex C6G/MCP/REST contracts and refusal packs.
6. Channel adapters: add ChatGPT, Claude, Gemini, WhatsApp, Telegram, API, MCP, and A2A adapters only after gate/session contracts exist.
7. Post-purchase UX: consume Grantex order, fulfillment, refund, support, dispute, settlement, and payout contracts only after Grantex implements them.

## Stop Conditions

- Any AgenticOrg commerce path calls payment providers, Plural, provider credential routes, merchant private APIs, or merchant existing systems directly.
- Any public A2A/MCP/channel metadata exposes commerce tools before Grantex and AgenticOrg gates agree.
- Any buyer response claims public discovery, checkout/payment availability, live provider access, live Plural access, production readiness, certification, compliance, conformance, standardization, merchant approval, or public-launch readiness.
- Any channel logs or displays raw passports, JWTs, provider credentials, webhook secrets, raw payloads, private merchant IDs, private tenant IDs, private URLs, production config, or allowlists.
- Any runtime implementation is needed to fix a matrix row; that must be a future PR, not hidden inside this docs slice.
